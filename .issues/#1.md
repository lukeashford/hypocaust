## 📝 Epic / Issue: **Scraper Agent MVP**

> **Goal:** Given a `ScrapeCompanyCommand` on Kafka, fetch the company’s public-facing web root, first-level social profiles, and any linked YouTube videos, then persist everything in Postgres so downstream organs can chunk & embed.
> **Target sprint:** 1 week (≈ 5 dev-days).
> **Owner:** `@scraper-team`
> **Reviewers:** `@orchestrator-team`, `@data-platform`

---

### 1. Context & constraints

* Each organ = **hexagonal service** (see architecture notes).
* **Storage:** **Postgres 16 + pgvector** only. No S3; raw artefacts go into DB for now.
* **Migrations:** **Liquibase**, XML changelogs.
* **Messaging:** Kafka (topics below).
* Licensed content & robots.txt compliance logged but not enforced yet.

### 3. Scope – *What the agent does*

1. **Receive command**

   ```json
   {
     "companyId": "UUID",
     "companyName": "nike",
     "homepage": "https://www.nike.com",
     "socialHandles": {
       "twitter": "nike",
       "youtube": "UC...ID"
     },
     "seedUrls": ["https://www.nike.com"]
   }
   ```

2. **Crawl logic**

  * Depth = 1 (homepage + child pages).
  * Respect `robots.txt`; skip if disallowed.
  * Fetch socials via official APIs **or** simple GET if unauthenticated endpoint.
  * For each YouTube link found: call `youtube-dl --dump-json` to pull metadata + transcript (if available).

3. **Persist**

  * Insert **`source_doc`** row per asset (see schema).
  * Insert **`raw_html` / `raw_json` / `raw_text`** into new `raw_content` TEXT column (CLOB OK).
  * Notify downstream by emitting `scrape.done`.

4. **Observability & retries**

  * **Micrometer** timers for fetch latency.
  * Retries (3×, exponential) on 5xx or network failure; poison message → `scrape.error`.

### 4. DB schema (Liquibase)

```xml
<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog" 
                   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
                   http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-4.8.xsd">

    <changeSet id="1.1-create-source-doc" author="scraper">
        <createTable tableName="source_doc">
            <column name="id" type="UUID" defaultValueComputed="gen_random_uuid()" primaryKey="true"/>
            <column name="company_id" type="UUID"/>
            <column name="crawl_job_id" type="UUID"/>
            <column name="source_type" type="VARCHAR(20)"/>
            <column name="external_id" type="VARCHAR(500)"/>
            <column name="url" type="VARCHAR(2000)"/>
            <column name="title" type="TEXT"/>
            <column name="raw_content" type="TEXT"/>
            <column name="lang_code" type="VARCHAR(5)"/>
            <column name="fetched_at" type="TIMESTAMP WITH TIME ZONE" defaultValueComputed="CURRENT_TIMESTAMP"/>
            <column name="checksum" type="VARCHAR(64)"/>
        </createTable>

        <createIndex tableName="source_doc" indexName="idx_source_doc_company">
            <column name="company_id"/>
        </createIndex>
    </changeSet>
</databaseChangeLog>
```

*(Liquibase auto-roll-back via `dropTable` on same changeSet).*

### 6. Acceptance Criteria

* ✅ Given a valid command, **>= 90 %** of URLs fetched, rows stored, `scrape.done` produced.
* ✅ Invalid URL or 4xx → agent retries up to 3×, then emits `scrape.error`.
* ✅ Unit tests > 80 % coverage on domain & application layers.
* ✅ Integration test passes in CI (GitHub Actions workflow).
* ✅ Liquibase migration applies cleanly against fresh PG in pipeline.
* ✅ Metrics visible: `scraper_fetch_latency`, `scraper_errors_total`.
